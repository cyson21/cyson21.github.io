# 한국어 포트폴리오 문구 감사

- 날짜: 2026-07-19
- 범위: 홈, 프로젝트 목록·상세, 경력, 웹 이력서, 공통 컴포넌트

## 참고 자료

- 원티드 이력서 가이드: 3~5년 차는 역할을 수행한 방법, 도입한 기술과 전략을 구체적으로 설명
  - https://blog.wantedlab.com/library/insight/resume-tip1
- 커리어리 포트폴리오 가이드: 프로젝트 목적, 기간, 역할·기여도와 성과를 기본 정보로 제시
  - https://careerly.co.kr/community/post/86684
- 잡코리아 취업 Q&A: 경력 프로젝트를 `프로젝트명(기간)`, `담당업무 및 성과`, `기술스택`으로 정리
  - https://www.jobkorea.co.kr/User/Qstn/AnswerWrite?qstnNo=139589
- 국내 개발자 포트폴리오 사례: `프로젝트 보기`, `주요 기능`, `기술 스택`, `문제·원인·해결·결과`처럼 바로 이해되는 항목명 사용
  - https://shiniseong.github.io/
  - https://allen8524.github.io/
  - https://developer-pino.info/
- OKKY 개발자 포트폴리오 논의: 변경 시점은 `최근 업데이트 시점`으로 표현
  - https://okky.kr/articles/1558141

## 판단 기준

1. 채용 담당자와 개발자가 별도 설명 없이 이해하는 항목명을 사용한다.
2. 프로젝트의 목적, 기간, 담당 역할, 문제 해결 과정과 결과를 먼저 보여준다.
3. 코드와 테스트는 신뢰를 보강하되 `근거`, `보호`, `검증`을 반복해 자체 용어처럼 만들지 않는다.
4. 프로젝트 목록에는 수정일보다 프로젝트 기간을 보여준다.
5. 기술 용어를 제외한 제목과 안내 문장은 자연스러운 한국어로 쓴다.
6. 한국어 항목명은 본문과 같은 sans 계열을 사용하고 코드·경로·테스트명만 mono 계열로 구분한다.

## 유지한 표현

- `상태 정합성`, `멱등 처리`, `회귀 테스트`, `Transactional Outbox`처럼 개발자가 실제로 사용하는 기술 용어
- 수치와 결과의 근거를 과장하지 않는 제한 사항
- 문제, 조치와 결과를 분리한 경력 사례 구조
