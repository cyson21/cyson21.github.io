# 공개 포트폴리오 톤앤매너

## 정체성

이 사이트는 채용 마케팅 랜딩페이지가 아니라 **운영 백엔드 엔지니어의 기술 사례 기록**이다. 추상적인 역량 선언보다 문제, 조치, 코드와 검증 결과를 먼저 보여준다.

## 정보 순서

1. 페이지 이름을 먼저 밝힌다.
2. 해결한 문제와 담당 역할을 설명한다.
3. 기술 선택과 이유를 비교한다.
4. 주요 구현과 관련 테스트를 연결한다.
5. 테스트 결과, 제외 범위와 향후 개선을 구분한다.

## 타이포그래피

- 본문은 18px, 모바일은 17px을 기준으로 한다.
- 보이는 텍스트는 13px 미만으로 내리지 않는다.
- 페이지 H1은 데스크톱 60px 이하, 모바일 34px 이상을 유지한다.
- 작은 영문 eyebrow 뒤에 거대한 추상 제목을 쌓지 않는다.
- 페이지 제목은 `프로젝트`, `경력`, `손찬양`처럼 대상을 직접 쓴다.
- 글자 간격은 0을 유지하고 한글은 단어 단위로 줄바꿈한다.

## 레이아웃

- 섹션은 카드가 아니라 구분선과 여백으로 나눈다.
- 카드 안에 카드를 넣지 않는다.
- 번호는 장식이 아니라 정보 순서에만 사용한다.
- 문제 지도, 검증표와 코드 근거는 평평한 표·기록 구조로 표현한다.
- 데스크톱과 모바일 모두 같은 정보 순서를 유지한다.

## 색과 그래픽

- 바탕은 저채도 회백색, 본문은 흑연색을 쓴다.
- 파랑은 탐색과 정보, 초록은 통과·수렴, 호박색은 제한, 빨강은 실패에만 쓴다.
- 그래픽은 프로젝트의 실제 상태 흐름과 검증 결과를 설명할 때만 사용한다.
- 장식용 그라디언트, 광원, 큰 그림자와 의미 없는 차트는 사용하지 않는다.

## 문장

- `무엇을 했다`보다 `어떤 문제를 어떻게 고쳐 무엇을 확인했는가`를 쓴다.
- 국내 경력기술서와 개발자 포트폴리오에서 익숙한 `담당 업무`, `성과`, `기술 스택`, `문제 해결`, `주요 구현`, `관련 테스트`를 우선 사용한다.
- `구현 근거`, `보호 규칙`, `연결 테스트`, `검증 수준`처럼 뜻을 한 번 더 해석해야 하는 자체 용어를 만들지 않는다.
- 프로젝트 목록에는 수정일 대신 프로젝트 기간을 표시한다.
- 명사만 이어 붙이지 말고 주어와 동작이 드러나는 짧은 문장으로 쓴다.
- 면접 안내, 읽는 순서 지시, 자기평가 점수는 쓰지 않는다.
- 검증하지 않은 수치와 운영 규모를 만들지 않는다.
- 실무 근거와 개인 프로젝트 근거를 섞지 않는다.

## 금지 패턴

- 아주 작은 보조문구와 70px 이상의 추상 제목 조합
- 같은 슬로건을 페이지마다 반복
- 영문 라벨만으로 정보 구조를 설명
- 한 화면에 여러 개의 둥근 카드와 배지 중첩
- 근거 없는 기술 숙련도 차트와 점수
- 모바일에서 설명문을 숨겨 정보가 줄어드는 구성
