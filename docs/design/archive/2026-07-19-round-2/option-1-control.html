Project Detail · 개인 프로젝트

StockRush

한정 재고 주문에서 실패와 중복이 발생해도 주문·재고·결제 상태가 모순 없이 마무리되는 흐름을 구현했습니다.

담당 역할
주문·재고·결제 서비스 경계, Saga·Outbox, 게이트웨이 인증과 장애 복구 흐름 직접 설계·구현

확인한 결과

  • 같은 재고 이벤트 2회 1회만 처리 · 중복 후속 이벤트 0건
  • 늦은 승인 무시 / 취소 주문 상태 유지
  • 5회 실패 FAILED / 실패 상태와 재처리 경로 보존

해결한 문제와 담당 역할

결제 실패, 이벤트 중복과 Kafka 중단이 주문 흐름 중간에 끼어들어도 세 서비스의 상태가 서로 모순되지 않아야 했습니다.

  • 01주문·재고·결제 서비스의 상태 경계와 보상 순서를 설계했습니다.
  • 02Saga·Outbox를 적용하고 중복 이벤트와 늦은 응답을 다루는 복구 흐름을 구현했습니다.
  • 03게이트웨이 인증부터 실패 상태 보존과 재처리 경로까지 직접 설계·구현했습니다.

기술 선택과 이유

Saga 상태 조율
서비스별 로컬 트랜잭션을 유지하면서 주문 흐름의 현재 단계와 보상 여부를 명시적으로 관리하기 위해 선택했습니다.
트랜잭션 Outbox
도메인 상태 변경과 발행 대상을 같은 트랜잭션에 기록해, Kafka가 중단돼도 전달해야 할 이벤트를 잃지 않도록 했습니다.

주요 구현과 관련 테스트

  • 01같은 재고 이벤트를 2회 전달해 1회만 처리되고 중복 후속 이벤트가 0건인지 확인했습니다.
  • 02취소 이후 도착한 늦은 승인을 무시하고 취소 주문 상태가 유지되는지 확인했습니다.
  • 03발행이 5회 실패하면 FAILED로 남고 실패 상태와 재처리 경로가 보존되는지 확인했습니다.

프로젝트 범위

실결제, 다중 리전, 운영 규모 부하와 장시간 브로커 장애는 포함하지 않았습니다.
